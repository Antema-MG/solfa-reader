1 Format .msolfa — En-tête

Chaque fichier .msolfa commence par 5 lignes de métadonnées obligatoires, dans cet ordre exact. Aucune n'est optionnelle. Si l'une manque, le parser refuse le fichier.

Key: G
Mesure: 6/4
Titre: Nom du morceau
Compositeur: Nom du compositeur
Tempo: 80 BPM

Mesurenum/denom (ex: 6/4, 3/4, 4/4)Le numérateur = nombre de temps par mesure. Pilote le parser.
TitreTexte libreNom du morceau. Affiché dans l'interface.

2 Format .msolfa — Corps musical

Après l'en-tête, le corps est structuré en blocs de mesures. Chaque bloc commence par un commentaire de repère, suivi de 4 lignes de voix.

// Mesure 1
S. | m : r : d : s : - : - |
A. | d : d : d : m : - : - |
T. | s : f : m : s : - : - |
B. | d : d : d : d : - : - |

// Mesures 3 & 4
S. | s : - : - : - : - : - | s.f : m : s : d' : - : - |
A. | d : - : - : - : - : - | m.r : d : m : m  : - : - |
T. | m : - : - : - : - : - | s   : s, : s : s  : - : - |
B. | d : - : - : - : - : - | d   : d. : d : d  : - : - |

Règles structurelles

Commentaire de repère — format strict : // Mesure X ou // Mesures X & Y. Ligne seule, rien d'autre. Repère de navigation uniquement, pas utilisé pour compter les temps.

4 voix obligatoires par bloc — dans l'ordre : S. (Soprano) · A. (Alto) · T. (Ténor) · B. (Basse). Si une voix manque, le parser bloque.

Délimiteur | — encadre un groupe de temps. Un groupe peut représenter une demi-mesure ou une mesure complète. Le parser accumule les groupes successifs jusqu'à atteindre le total de la mesure (valeur du numérateur de Mesure).

Délimiteur : — sépare les temps à l'intérieur d'un groupe. Nombre de temps = nombre de : + 1. En 6/4, deux groupes de 3 temps | t:t:t | t:t:t | forment une mesure complète.

Bloc multi-mesures — une ligne peut contenir plusieurs groupes | ... | côte à côte. Chaque groupe est validé séparément.

3 Format .msolfa — Symboles rythmiques

SymboleExempleDuréeDescription
Note seulem1.0 tempsAttaque + tenue sur tout le temps.
Tiret-1.0 tempsProlongation. Pas de nouvelle attaque. Tenue de la note précédente.
Silence0 ou _1.0 tempsSilence complet. L'oscillateur est coupé pour ce temps.
Deux crochess.f1.0 tempsDeux attaques : première note 0.5 + deuxième note 0.5. Le point est entre les deux notes.
Note pointéed.1.0 tempsAttaque sur 0.5, tenue sur 0.5. Le point est après la note. Pas de deuxième attaque.

Distinction critique : d. = 1 temps total (note pointée dans un temps). d : - = 2 temps (note + prolongation). Ne pas confondre.

4 Format .msolfa — Hauteurs et modificateurs

Les 7 degrés de la gamme majeure : d (Do) · r (Ré) · m (Mi) · f (Fa) · s (Sol) · l (La) · t (Si). Degrés relatifs — la hauteur absolue dépend de la tonique active.

Modificateurs d'octave — collés à la note, sans espace

ModificateurExempleEffet
'd'Octave supérieure (+12 demi-tons)
''d''Double octave supérieure (+24 demi-tons)
,s,Octave inférieure (−12 demi-tons)
,,s,,Double octave inférieure (−24 demi-tons)
ifi, si, ri, diDemi-ton haut (dièse). Collé après la lettre, avant l'octave.

Interdit : espace entre la note et son modificateur. d ' est invalide. Seul d' est correct.

5 Règles de validation — Erreurs fréquentes

// ERREUR — trop de temps en 3/4 (4 temps au lieu de 3)
S. | d : r : m : s |

// ERREUR — modificateur avec espace (espace avant apostrophe)
S. | d ' : r : m |

// ERREUR — voix manquante (T. et B. absents)
S. | d : r : m |
A. | s : l : t |

// ERREUR — métadonnée manquante (Titre, Compositeur, Tempo absents)
Key: G
Mesure: 4/4

// ERREUR — note et croche mixées incorrectement
S. | s. : r : m |  ← point après note seule sans deuxième note

Le parser lève une erreur bloquante avec message précis : voix concernée, numéro de mesure, total reçu, total attendu. Exemple : "Erreur Alto Mesure 4 : Reçu 2.5 temps au lieu de 3".

7 Parser M-Solfa — Fonctionnement

Le parser est la pièce centrale. Il transforme le texte brut en structure de données typée consommée par le moteur audio et l'interface. Il doit être entièrement testé avec Vitest avant de passer au reste.

Étape 1
Tokenisation (Lexer)

Chaque ligne est classée : METADATA, COMMENT, VOICE_LINE, EMPTY_LINE. Le lexer ne valide pas, il découpe.

Étape 2
Validation métadonnées

Présence des 5 clés obligatoires. Format Key (regex A-G), Mesure (num/denom), Tempo (entier BPM).

Étape 3
Parse des voix

Découpe chaque ligne par | puis par :. Identifie chaque token comme note / tiret / silence / croche / pointée.

Étape 4
Validation du compte

Total des durées par groupe vs numérateur de Mesure. Si écart : erreur bloquante avec voix + mesure + reçu + attendu.

Étape 5
Cohérence inter-voix

Toutes les 4 voix d'un même bloc doivent avoir le même nombre de mesures reconstituées.

Étape 6
Résultat typé

Retourne ParseResult : succès avec MsolFile ou échec avec tableau de ParseError[]. Jamais les deux.

8 Moteur audio — Tone.js

Synthèse 100% temps réel via Web Audio API / Tone.js. Aucun fichier son externe. Fonctionne hors ligne après chargement de la page.

4 timbres synthétisés

TimbreOscillateurEnveloppeEffets
OrgueSinusoïdal (5 harmoniques)Attack 20ms, Sustain 1.0, Release 50msTone.Reverb decay:3 wet:0.4
PianoTriangulaireAttack 5ms, Decay 300ms, Sustain 0.1Aucun — son percussif naturel
CordesDents de scieAttack 200ms, Sustain 0.9, Release 300msFilter lowpass 2000Hz + Chorus
CuivresCarré (square)Attack 30ms, Sustain 0.8, Release 100msFilter lowpass 3000Hz

Architecture de chaîne audio par voix

Oscillateur(s)
  → GainNode (enveloppe ADSR)
  → Filter (selon timbre)
  → VolumeNode (mute / solo)
  → VolumeNode global
  → AudioContext.destination

Calcul des fréquences — Tonal.js

Intervalles demi-tons des degrés : d=0 r=2 m=4 f=5 s=7 l=9 t=11. Chromatic i = +1. Octave ' = +12, , = −12. Formule MIDI → Hz : f = 440 × 2^((midi−69)/12).

Scheduling audio

Toutes les notes sont schedulées à l'avance via oscillator.start(audioContext.currentTime + offset). L'horloge maître est AudioContext.currentTime — pas un setInterval. Garantit zéro drift. Quand le tempo change, les offsets restants sont recalculés depuis la position courante.

Mute / Solo

Mute : gain de la voix mis à -Infinity avec ramp 5ms (évite les clics audio). Solo : toutes les autres voix mutées. Désactivation : gain restauré à 0 dB avec le même ramp.

10 Partition défilante — Détails

Rendu
Affichage propre

Les 4 lignes de voix alignées verticalement par colonne de temps. Les commentaires // affichés comme titres de section. Pas de balises techniques visibles.

Sync
Surlignage temps réel

Boucle requestAnimationFrame lisant AudioContext.currentTime. La colonne de temps active est mise en évidence simultanément sur les 4 voix. Jamais en avance ni en retard sur l'audio.

Navigation
Barre de progression

Cliquable. Un clic repositionne la tête de lecture audio et le surlignage instantanément. Le moteur audio reprend depuis ce point.

Sélection
Section loop

Cliquer-glisser sur la partition sélectionne une plage. La lecture se restreint à cette plage (arrêt en fin ou boucle). Désélection = retour global.

11 Piano virtuel — Détails

Visuel du clavier

Soprano (rouge)
Alto (bleu)
Ténor (vert)
Basse (orange)

Étendue fixe : 43 touches, Do2 (C2) à Sol5 (G5). Les touches blanches et noires aux proportions réelles. L'étendue ne change pas avec la transposition — c'est l'étendue physique. Les touches qui s'illuminent changent selon la tonique active.

Règle absolue du 3+1 mains

Main droite — 3 doigts actifs
Soprano — doigt le plus haut. Registre aigu.
Alto — doigt médian. Registre moyen.
Ténor — pouce. Basculé automatiquement à l'octave supérieure si trop grave. Toujours dans la zone main droite.
Main gauche — 1 seul doigt
Basse — une unique touche dans le registre grave. Jamais un accord. Jamais deux touches simultanées.
La main gauche marque toujours la note fondamentale seule. C'est une règle absolue — aucune exception.

Bascule automatique du Ténor

Si la note réelle du Ténor tombe en dessous du Do3, le système la transpose automatiquement une octave vers le haut pour la maintenir dans la zone jouable de la main droite. Ce décalage est uniquement visuel (affichage piano) — la note jouée par le moteur audio n'est pas modifiée.

Mode apprentissage (Pause)

Pause + clic sur un temps

Le moteur audio se tait. Le piano maintient allumées les 4 touches (3 droite + 1 gauche) indéfiniment. L'utilisateur peut poser ses mains sur un vrai piano à son rythme.

Bandeau d'aide textuelle

Juste au-dessus du piano. Affiche en temps réel : Main Gauche (1 doigt) : [ Sol ] | Main Droite (3 doigts) : [ Mi - Do - Sol ]. Mis à jour à chaque temps. Figé au dernier temps cliqué en mode pause.

Filtre Solo sur le piano

Si une voix est en Solo, seules les touches de cette voix s'illuminent sur le clavier. Les touches des autres voix restent éteintes. Le bandeau d'aide mentionne uniquement la/les voix solos.

12 Contrôles — Tous les features

Transport
Lecture / Pause / Stop

Lecture démarre ou reprend. Pause gèle à l'instant précis, maintient l'état visuel. Stop remet tout à zéro (position, surlignage, piano).

Transport
Tempo en temps réel

Slider ou champ numérique. Plage 20–240 BPM. Modifiable pendant la lecture sans interruption. Le moteur recalcule les timestamps restants.

Transposition
Sélecteur de tonique

Do, Ré, Mi, Fa, Sol, La, Si, Sib, Mib, Lab, Reb, Fa#. La transposition est appliquée instantanément. Affichage mis à jour : "Clé de Ré". Le fichier source n'est pas modifié.

Audio
Sélecteur de timbre

Orgue, Piano, Cordes, Cuivres. Changeable en temps réel. Les nouvelles attaques utilisent le nouveau timbre. Global pour les 4 voix.

Voix
Mute individuel

Un bouton par voix (S, A, T, B). Coupe le son sans interrompre la lecture. L'oscillateur continue pour maintenir la synchronisation.

Voix
Solo

Un bouton par voix. Isole une ou plusieurs voix. Les autres sont mutées. Impacte aussi le piano (seules les touches de la voix solo s'illuminent).

Navigation
Barre de progression

Cliquable n'importe où. Repositionne instantanément la tête de lecture audio et le surlignage visuel.

Navigation
Sélection de section

Cliquer-glisser sur la partition pour sélectionner une plage. La lecture se restreint à cette plage. Boucle ou arrêt en fin de sélection.

Fichiers
Ouverture de fichier

File System Access API (boîte de dialogue native) ou drag-and-drop. Lecture du contenu texte. Si erreur de parsing, message précis affiché. Ancien morceau conservé.

Persistance
Préférences locales

localStorage pour timbre préféré, tempo par défaut, tonique préférée. IndexedDB pour l'historique des fichiers récents et leur contenu en cache.

14 Types TypeScript — Domaine M-Solfa

// Degrés de la gamme
type Degree = 'd' | 'r' | 'm' | 'f' | 's' | 'l' | 't'

// Modificateurs d'octave
type OctaveModifier = '' | "'" | "''" | ',' | ',,'

// Voix
type Voice = 'S' | 'A' | 'T' | 'B'

// Modificateur chromatique
type ChromaticModifier = 'i' | null

// Élément rythmique
interface MsolNote {
  type: 'note'
  degree: Degree
  octave: OctaveModifier
  chromatic: ChromaticModifier
  duration: number        // 1.0 ou 0.5
  isSecondHalf?: boolean  // true si 2ème croche d'un s.f
}
interface MsolRest  { type: 'rest'; duration: number }
interface MsolTie   { type: 'tie';  duration: number }
type MsolElement = MsolNote | MsolRest | MsolTie

// Un temps (1.0 durée totale)
interface MsolBeat {
  elements: MsolElement[]
  totalDuration: number  // toujours 1.0
}

// Une mesure complète
interface MsolMeasure {
  index: number
  beats: Record<Voice, MsolBeat[]>
  totalBeats: number
}

// Résultat du parsing
type ParseResult =
  | { success: true;  file: MsolFile }
  | { success: false; errors: ParseError[] }

15 Roadmap de développement

Phase 0 — Environnement (1 jour)
Node 20, Git, VS Code. npx create-next-app@latest avec TypeScript + Tailwind + App Router + src dir. Installation de toutes les dépendances. Configuration tsconfig strict. Setup Vitest. Création de la structure des dossiers.
Phase 1 — Types + Parser + Tests (3–4 jours)
Fichier msolfa.types.ts complet. Lexer, Validator, Parser. Tests Vitest exhaustifs : mesure valide, métadonnée manquante, mauvais compte de temps, croches, modificateurs d'octave. Ne pas continuer sans tous les tests verts.
Phase 2 — Moteur audio (3–4 jours)
Module transposition.ts avec Tonal.js. Les 4 timbres (organ, piano, strings, brass). Engine.ts avec MsolfaAudioEngine. Scheduling Tone.js. Mute/Solo. Test dans la console navigateur.
Phase 3 — Store Zustand (1 jour)
Store global msolfa.store.ts avec tous les états et actions. Colle le parser, l'audio et l'interface.
Phase 4 — Interface de base (2–3 jours)
Page unique (page.tsx). FileOpener.tsx (ouverture fichier local). Transport.tsx (Lecture/Pause/Stop + BPM). VoiceControls.tsx (Mute/Solo SATB). MetadataBar.tsx. Premier rendu fonctionnel.
Phase 5 — Partition défilante (2–3 jours)
ScrollingScore.tsx. Rendu propre du texte .msolfa. Boucle requestAnimationFrame. Surlignage synchronisé sur AudioContext.currentTime. Barre de progression cliquable.
Phase 6 — Piano virtuel (3–4 jours)
VirtualPiano.tsx avec react-piano. Mapping degrés → MIDI → touches. Règle 3+1 mains (handRouter.ts). Illumination couleurs SATB. Mode apprentissage pause. HelpBanner.tsx (bandeau aide textuelle). Filtre Solo.
Phase 7 — Polish & Persistance (2–3 jours)
Sélecteur tonique + transposition à la volée. Sélection de section (loop). localStorage préférences. IndexedDB historique fichiers. Tests finaux. Vérification dark mode.

Règle absolue du développement : Ne jamais passer à la phase suivante sans que la phase courante soit fonctionnelle et testée. Le parser (Phase 1) est la fondation de tout — s'il est buggé, tout le reste l'est aussi.
